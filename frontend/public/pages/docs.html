How BrewLine is built.

An end‑to‑end walkthrough — architecture, technology, implementation, testing, AI usage and future improvements.

1 · Introduction

The dairy of single‑serve coffee is brewed in capsules: tiny aluminium / plastic cups that contain a precisely dosed bed of ground coffee, sealed with a foil lid. Factories produce these at very high speeds — typically 8–25 capsules per minute per head — under tight tolerances. This project simulates such a line and shows how an operator would monitor it through a modern HMI / SCADA stack.

2 · Objectives

  • Reproduce the full five‑stage production flow visually.
  • Provide industrial controls: Start / Stop / Reset / Emergency Stop.
  • Classify defects into five distinct failure modes with traceable IDs.
  • Operate a CMMS module that degrades machine health during runtime.
  • Stream metrics in InfluxDB line protocol for Grafana consumption.
  • Deliver a static bundle deployable to GitHub Pages.

3 · System architecture

┌──────────────────┐   ┌──────────────┐   ┌──────────────┐   ┌─────────────┐
│  Browser (HMI)   │←→ │  Simulator   │ → │  InfluxDB v2 │ → │   Grafana   │
│  HTML / CSS / JS │   │  Python · JS │   │  bucket      │   │  dashboard  │
└──────────────────┘   └──────────────┘   └──────────────┘   └─────────────┘
        ▲                       │
        └── shared localStorage ┘   (in‑browser, GitHub Pages mode)

The browser ships a self‑contained simulator that ticks at 1 Hz, persists state in localStorage, and renders multiple synchronised views (Production, SCADA, Quality, CMMS, Analytics). A parallel Python simulator (in /simulation/) reproduces the same algorithm against InfluxDB to demonstrate real‑plant deployment.

4 · Technology stack

  • Frontend — semantic HTML5, modern CSS (custom properties, grid, container queries), vanilla ES2022. No framework — every byte is reviewable.
  • Simulation — Python 3.11 with influxdb-client for the offline replica.
  • Time series — InfluxDB v2 (bucket = brewline, retention 30d).
  • Dashboards — Grafana 10 with a pre‑provisioned JSON.
  • Tooling — VS Code, Git, GitHub Pages.

5 · Implementation

The shared simulator lives in assets/js/main.js as a Brew singleton. It exposes a clean API:

Brew.start()           // begin production
Brew.stop()            // pause
Brew.eStop()           // emergency stop
Brew.releaseEStop()    // clear E-Stop
Brew.reset()           // zero counters & logs
Brew.setSpeed(cpm)     // capsules / minute
Brew.setDefectRate(r)  // 0 .. 0.5
Brew.triggerDefect(k)  // inject a specific defect

Every page subscribes via Brew.subscribe(fn) and re‑renders when state changes. The conveyor on the Production page is a pure DOM animation — capsules are absolutely positioned divs whose left coordinate is advanced each frame in a requestAnimationFrame loop.

6 · Testing

  • Manual UI verification across the 8 pages on desktop & mobile breakpoints.
  • Defect classifier tested by triggering each of the 5 failure modes and verifying counters & history.
  • E‑Stop verified to block start() until explicitly released.
  • InfluxDB writer validated locally with influx query 'from(bucket:"brewline")|>range(start:-1h)'.
  • Grafana dashboard imported and visually checked against panel definitions.

7 · AI usage

This project was authored with assistance from an AI coding agent. The agent was used to scaffold boilerplate, accelerate styling and propose the defect classifier weights. All algorithmic decisions, the architectural shape and the integration choices were reviewed and validated by the author. AI‑generated code was tested for correctness before inclusion.

8 · Future improvements

  • WebSocket bridge so the static UI consumes real InfluxDB live (instead of localStorage).
  • Computer‑vision QC: replace the random classifier with a TensorFlow.js model trained on capsule defect images.
  • OPC‑UA gateway for connecting BrewLine to a Siemens / Allen‑Bradley PLC.
  • Multi‑line tenancy & recipe management.

9 · Deployment to GitHub Pages

git init
git add .
git commit -m "1 Setup"
git branch -M main
git remote add origin https://github.com/<you>/brewline.git
git push -u origin main

# Then in Settings → Pages → Source: deploy from `main` / `/ (root)`.
